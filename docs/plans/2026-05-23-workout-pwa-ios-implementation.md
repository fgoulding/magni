# Magni - iOS-Centric Implementation Plan

> **For Claude:** REQUIRED SUB-SKILL: Use superpowers:executing-plans to implement this plan task-by-task.

**Goal:** Build a self-hosted, multi-user, iPhone-first workout planner that runs well in Safari and as an installed iOS Home Screen web app.

**Architecture:** Next.js current stable App Router + TypeScript, SQLite via better-sqlite3, TailwindCSS current stable, scoped service worker, httpOnly cookie sessions, Docker deployment with a bind-mounted SQLite database.

**Design doc:** `docs/plans/2026-05-23-workout-pwa-ios-design.md`

## Current Technology Choices

Use current stable versions at implementation time unless a compatibility issue appears during scaffolding.

| Area | Default | Alternative | Notes |
|------|---------|-------------|-------|
| Web framework | Next.js current stable App Router | Remix, TanStack Start | Keep route handler params aligned with selected Next version. Do not mix Next 14 examples with Next 15/16 route signatures. |
| Styling | TailwindCSS current stable | CSS Modules, Panda CSS | Tailwind v4 is current. Use the official Next/Tailwind setup created by `create-next-app`. |
| DB | SQLite + better-sqlite3 | Drizzle ORM on SQLite, Prisma | Raw SQL is fine for this app. Drizzle is the best next step if query safety becomes painful. |
| Password hashing | Argon2id | bcryptjs / bcrypt | Prefer Argon2id per OWASP. Use bcrypt only if Argon2 native/package constraints get awkward in Docker. |
| PWA tooling | Manual manifest + service worker | Serwist / Workbox | Manual is enough for V1 if caching stays simple. Use Serwist/Workbox if runtime caching grows. |
| iOS native path | PWA | Capacitor | Capacitor is the recommended upgrade path if native iOS features become required. |

## Important iOS Constraints

- Do not depend on a browser install prompt. Add an Install Help screen with Safari Share -> Add to Home Screen instructions.
- Do not rely on Background Sync for workout saves. iOS support is not a solid V1 foundation.
- Do not cache authenticated API responses in the service worker.
- Do not use placeholder 1x1 icons. Generate real 192, 512, and Apple touch icons.
- Push notifications are possible only for installed Home Screen apps on supported iOS versions, but are out of V1 scope.
- The server database is the source of truth. Any local pending state is a temporary retry aid, not durable offline storage.

## Fixed Issues From Review

- Use one Next.js version model consistently. If using Next 15/16, route params are Promise-shaped; if using Next 14, they are synchronous. The implementation must follow the scaffolded version.
- Replace `next lint` with the current Next/ESLint setup generated by `create-next-app`.
- Next.js proxy middleware does not attach `req.user`; API handlers validate sessions with `getUser()` / `requireUser()`.
- Logout deletes the server session row and clears the cookie.
- Store `progression_type` and `auto_progression_enabled` on exercises.
- Apply TM progression exactly once during a transactional `complete-and-advance` operation.
- Rename `actual_weight_used` / `actual_weight` consistently to `actual_weight`.
- Service worker caches app-shell/static assets only and excludes API requests.
- UI tasks must be fully specified before implementation; no "same pattern" placeholder steps.

## Task 2: Database Schema

**Files:** `src/lib/db/schema.sql`, `src/lib/db/index.ts`, `src/lib/db/__tests__/db.test.ts`

Create schema with these tables:

```sql
CREATE TABLE IF NOT EXISTS users (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  email TEXT NOT NULL UNIQUE,
  password_hash TEXT NOT NULL,
  created_at TEXT NOT NULL DEFAULT (datetime('now'))
);

CREATE TABLE IF NOT EXISTS auth_sessions (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  user_id INTEGER NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  token TEXT NOT NULL UNIQUE,
  expires_at TEXT NOT NULL,
  created_at TEXT NOT NULL DEFAULT (datetime('now'))
);

CREATE TABLE IF NOT EXISTS programs (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  user_id INTEGER NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  name TEXT NOT NULL,
  description TEXT NOT NULL DEFAULT '',
  num_weeks INTEGER NOT NULL DEFAULT 7 CHECK(num_weeks > 0),
  current_week INTEGER NOT NULL DEFAULT 1 CHECK(current_week > 0),
  current_day INTEGER NOT NULL DEFAULT 1 CHECK(current_day > 0),
  created_at TEXT NOT NULL DEFAULT (datetime('now'))
);

CREATE TABLE IF NOT EXISTS days (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  program_id INTEGER NOT NULL REFERENCES programs(id) ON DELETE CASCADE,
  name TEXT NOT NULL,
  day_number INTEGER NOT NULL CHECK(day_number > 0),
  sort_order INTEGER NOT NULL DEFAULT 0,
  UNIQUE(program_id, day_number)
);

CREATE TABLE IF NOT EXISTS exercises (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  day_id INTEGER NOT NULL REFERENCES days(id) ON DELETE CASCADE,
  name TEXT NOT NULL,
  training_max REAL NOT NULL CHECK(training_max > 0),
  category TEXT NOT NULL DEFAULT 'main' CHECK(category IN ('main','aux','accessory')),
  progression_type TEXT NOT NULL DEFAULT 'custom',
  auto_progression_enabled INTEGER NOT NULL DEFAULT 0,
  sort_order INTEGER NOT NULL DEFAULT 0
);

CREATE TABLE IF NOT EXISTS week_settings (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  exercise_id INTEGER NOT NULL REFERENCES exercises(id) ON DELETE CASCADE,
  week_number INTEGER NOT NULL CHECK(week_number > 0),
  intensity_pct REAL NOT NULL CHECK(intensity_pct >= 0 AND intensity_pct <= 1),
  reps INTEGER NOT NULL CHECK(reps > 0),
  sets INTEGER NOT NULL CHECK(sets > 0),
  rep_out_target INTEGER NOT NULL CHECK(rep_out_target >= 0),
  calculated_weight REAL,
  UNIQUE(exercise_id, week_number)
);

CREATE TABLE IF NOT EXISTS sessions (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  program_id INTEGER NOT NULL REFERENCES programs(id) ON DELETE CASCADE,
  user_id INTEGER NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  day_id INTEGER NOT NULL REFERENCES days(id) ON DELETE CASCADE,
  week_number INTEGER NOT NULL CHECK(week_number > 0),
  completed INTEGER NOT NULL DEFAULT 0,
  completed_at TEXT,
  date TEXT NOT NULL DEFAULT (date('now')),
  UNIQUE(program_id, user_id, day_id, week_number, date)
);

CREATE TABLE IF NOT EXISTS session_sets (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  session_id INTEGER NOT NULL REFERENCES sessions(id) ON DELETE CASCADE,
  week_setting_id INTEGER NOT NULL REFERENCES week_settings(id) ON DELETE RESTRICT,
  actual_reps INTEGER,
  actual_weight REAL,
  tm_delta_applied REAL NOT NULL DEFAULT 0,
  notes TEXT NOT NULL DEFAULT '',
  UNIQUE(session_id, week_setting_id)
);

CREATE TABLE IF NOT EXISTS user_settings (
  user_id INTEGER NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  key TEXT NOT NULL,
  value TEXT NOT NULL,
  PRIMARY KEY (user_id, key)
);
```

Implementation notes:

- Use WAL mode and `PRAGMA foreign_keys = ON`.
- Use `DB_PATH`, defaulting to `data/workouts.db`.
- Tests must use a test database path, not the developer's real local data file.
- Test table creation, unique email, user-scoped program rows, and session set uniqueness.

Proxy middleware responsibilities:

- Allow `/api/auth/*`, static assets, manifest, service worker, icons, and public install assets.

- Return 401 for unauthenticated API requests.
- Do not pretend to attach `req.user`.
- Database validation remains in route handlers.

Tests:

- Password hash verifies the right password and rejects the wrong one.
- Tokens are unique 64-character hex strings.
- Deleting a session invalidates `getUser()` for that token.

## Task 7: Sessions And Idempotent Completion

**Files:**

- `src/app/api/programs/[id]/sessions/route.ts`
- `src/app/api/sessions/route.ts`
- `src/app/api/sessions/[sessionId]/sets/route.ts`
- `src/app/api/programs/[id]/complete-and-advance/route.ts`

Implement:

- `POST /api/programs/:id/sessions`: create or return today's uncompleted session for the program's current week/day.
- `GET /api/programs/:id/sessions`: list program sessions.
- `GET /api/sessions`: global history for current user.
- `PUT /api/sessions/:sessionId/sets`: update reps, weight, and notes only. Do not change TM here.
- `POST /api/programs/:id/complete-and-advance`: transactional completion.

Completion transaction:

1. Load program, session, sets, week settings, and exercises scoped to current user.
2. If `session.completed = 1`, return the existing completed state without reapplying TM deltas.
3. For each exercise with `auto_progression_enabled = 1`, calculate the TM delta from the logged AMRAP set.
4. Store the per-set `tm_delta_applied`.
5. Update exercise `training_max`.
6. Recalculate future week `calculated_weight` values.
7. Mark session completed with `completed_at`.
8. Advance to next day/week, wrapping after the final week.

Tests:

- Updating reps repeatedly does not change TM.
- Completing once changes TM once.
- Calling complete twice does not apply a second delta.
- Custom progression never changes TM.
- User A cannot complete User B's session.
